## 二、测试执行规范

### 2.1 执行前准备

#### 2.1.1 环境检查清单

```
□ 后端服务已启动，端口8080可访问
□ 前端服务已启动，端口3000可访问
□ MySQL数据库已连接，测试数据已初始化
□ 测试账号已创建（普通用户/管理员/门店管理员）
□ 浏览器已清理缓存
□ 测试工具已准备（Postman/JMeter等）
```

#### 2.1.2 测试数据准备

| 数据类型 | 准备要求 |
|---------|---------|
| 测试用户 | 至少3个角色：普通用户、管理员、门店管理员 |
| 测试车辆 | 至少5辆不同状态的车辆 |
| 测试订单 | 覆盖各种订单状态 |
| 测试门店 | 至少2个门店 |

### 2.2 用例执行规范

#### 2.2.1 执行原则

1. **顺序执行**：按照用例编号顺序执行，不得跳过
2. **独立执行**：每个用例独立执行，不依赖其他用例结果
3. **一次一条**：每次只执行一条用例，便于问题定位
4. **即时记录**：执行后立即记录结果，不得事后补录

#### 2.2.2 执行状态定义

| 状态 | 定义 | 处理方式 |
|------|------|---------|
| 通过 | 实际结果与预期结果一致 | 标记通过，继续下一条 |
| 失败 | 实际结果与预期结果不一致 | 记录缺陷，标记失败 |
| 阻塞 | 前置条件不满足，无法执行 | 标记阻塞，说明原因 |
| 跳过 | 用例暂时不需要执行 | 标记跳过，说明原因 |
| 不适用 | 用例在当前版本不适用 | 标记N/A，说明原因 |

#### 2.2.3 缺陷记录规范

发现缺陷时必须记录以下信息：

```
缺陷标题：[模块名称] + [简要描述]
缺陷描述：
  - 前置条件：
  - 操作步骤：
  - 实际结果：
  - 预期结果：
严重程度：致命/严重/一般/轻微
优先级：紧急/高/中/低
截图/录屏：必须附加
环境信息：浏览器版本、操作系统
```

### 2.3 模块测试流程

#### 2.3.1 模块测试步骤

```
Step 1: 模块准备（30分钟）
  □ 阅读模块需求文档
  □ 确认测试环境就绪
  □ 准备测试数据
  □ 创建模块测试任务

Step 2: 用例执行（按模块规模）
  □ 按顺序执行所有用例
  □ 实时记录执行结果
  □ 发现缺陷立即记录

Step 3: 缺陷跟踪（持续进行）
  □ 提交缺陷到缺陷管理系统
  □ 跟踪缺陷修复进度
  □ 验证修复结果

Step 4: 回归测试（缺陷修复后）
  □ 执行失败用例的回归测试
  □ 执行相关联用例的回归测试
  □ 确认无新增缺陷

Step 5: 模块验收（模块完成后）
  □ 统计模块测试数据
  □ 输出模块测试报告
  □ 模块负责人签字确认
```

#### 2.3.2 模块验收标准

| 验收项 | 标准 |
|-------|------|
| 用例执行率 | 100%（所有用例至少执行一次） |
| 用例通过率 | ≥95%（P0用例必须100%通过） |
| 缺陷关闭率 | ≥90%（严重缺陷必须全部关闭） |
| 遗留缺陷 | P0/P1缺陷必须为0 |

## 五、测试管理规范

### 5.1 每日站会

**时间**：每天上午9:30
**时长**：15分钟
**参与人**：测试团队、开发代表
**内容**：
- 昨天完成的工作
- 今天计划的工作
- 遇到的问题和风险

### 5.2 缺陷管理流程

```
发现缺陷 → 提交缺陷 → 分配缺陷 → 修复缺陷 → 验证缺陷 → 关闭缺陷
    ↑___________________________________________________________↓
                          验证不通过
```

**缺陷状态流转**：
- 新建 → 已分配 → 已修复 → 已验证 → 已关闭
- 已验证 → 重新打开（验证不通过）
- 新建 → 已拒绝（不是缺陷）

### 5.3 测试暂停/恢复标准

**暂停标准**：
- 出现阻塞性缺陷，导致大量用例无法执行
- 测试环境故障，无法恢复
- 需求重大变更

**恢复标准**：
- 阻塞性缺陷已修复
- 测试环境已恢复
- 需求变更已确认并更新
